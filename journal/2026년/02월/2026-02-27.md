## ⚠️ Trouble
- **문제** : 프로젝트 기능 명세를 작성하면서 `사용자 스토리 - 전제 조건 - 메인 플로우 - 상태/데이터 변화 - API/데이터 계약 - 화면 - UI 변화 - 예외 - 테스트 시나리오`와 같은 상세한 구조를 적용하려다 보니 명세 작성 시간이 너무 오래 걸렸다.
- **원인** : 완벽한 명세를 한 번에 완성하려는 욕심이 작성 속도를 늦추는 원인이 되었다.
- **해결** : 명세 범위를 조정하기로 했다. 자세한 화면 설계와 시스템 관점 요소는 제외하고, 각 기능에 대해 핵심 동작 흐름과 예외 케이스에 집중해 정리했다.
- **리마인드** : 처음부터 완벽한 명세를 만들기보다 우선 개발이 가능한 수준으로 정리하고 점진적으로 다듬는 방식이 더 효율적이라는 것을 배웠다.
